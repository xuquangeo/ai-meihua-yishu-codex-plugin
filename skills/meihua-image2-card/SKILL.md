# 梅花易数 image2 教学卡

这个 Skill 单独负责：

- 把已经算好的梅花易数结果，最终画成一张 `image2 / image_gen` 教学卡

它不是计算 Skill。
它也不是普通 SVG 可视化 Skill。
它负责的是插件里的“最终展示版”。

## 前台输出硬规则

- 最终给用户看的主结果，固定为 `PNG`
- 这张 `PNG` 必须通过 `image2 / image_gen` 生成
- 不要把 `SVG` 返回为默认主结果
- 如果当前链路里同时存在 `SVG` 和 `PNG`，用户优先看到的必须是 `PNG`

它默认接收来自引导层已经整理好的输入摘要：

- 问题
- 数字

## 必读参考

先读取：

- `../../docs/image2-teaching-card-spec.md`
- `../../docs/methodology-reference.md`
- `../../docs/reference-case-image2-card-rain-73.md`
- `../../docs/trigram-visual-reference.md`

## 事实来源

教学卡里的事实必须来自：

- `result.json`
- `qa_report.json`
- `upper_trigram.svg`
- `lower_trigram.svg`
- `primary_hexagram.svg`
- `changed_hexagram.svg`
- `step_by_step_report.svg`

其中第二步“本卦”必须优先服从 `result.json -> primary -> assembly_lock`，
按这个锁定信息直接拼接，不允许自行重绘上半和下半。

第三步“动爻变化”必须优先服从：

- `qa_report.json -> expected -> moving_line`
- `qa_report.json -> expected -> moving_line_from_top`
- `qa_report.json -> expected -> moving_line_change`

禁止让 image2 自己猜：

- 卦名
- 动爻
- 体用
- 五行
- 最终判断

## 八卦图示硬规则

图示不能只靠卦名猜，必须按固定规则画：

- `阳` = 一整横
- `阴` = 中间断开的两短横

八卦从上到下固定是：

- 乾：阳 / 阳 / 阳
- 兑：阴 / 阳 / 阳
- 离：阳 / 阴 / 阳
- 震：阴 / 阴 / 阳
- 巽：阳 / 阳 / 阴
- 坎：阴 / 阳 / 阴
- 艮：阳 / 阴 / 阴
- 坤：阴 / 阴 / 阴

如果卦名与图示不一致：

- 判定为错误图
- 不要交付

并且数量也必须对：

- 每个八卦一定是 `3` 条爻线
- 每个六爻卦一定是 `6` 条爻线
- 不能把相邻两爻合并成一条粗线
- 不能把三爻卦偷画成两条

## 目标产物

默认输出：

- `image2_teaching_card.png`

定位是：

- 比 SVG 更好看
- 比普通海报更像教材
- 保留四步推演结构
- 保留第二步“数字 -> 八卦 -> 五行”
- 保留第四步“五行生克 + 体用判断”

## 固定结构

必须沿用当前插件已经跑成功的版式方向：

1. 明确问题，给数字
2. 做计算，算本卦
3. 做计算，算变卦
4. 做解析

## 基础知识区必须体现

教学卡里必须单独保留一块“八卦图形对照区”。

这块区域不是装饰，而是后续推演要反复引用的基础知识。

至少要列出：

- 乾：阳 / 阳 / 阳
- 兑：阴 / 阳 / 阳
- 离：阳 / 阴 / 阳
- 震：阴 / 阴 / 阳
- 巽：阳 / 阳 / 阴
- 坎：阴 / 阳 / 阴
- 艮：阳 / 阴 / 阴
- 坤：阴 / 阴 / 阴

这块区域建议放在第二步附近，或放在整张图右侧作为固定参考区。

## 第二步必须体现

- 第一位数字如何对应上卦
- 第二位数字如何对应下卦
- `1-8 -> 八卦`
- `八卦 -> 五行`
- 本例里数字到八卦、八卦到五行的链路
- 并直接引用“八卦图形对照区”来说明为什么这个数字对应这个卦形

## 本卦图硬规则

在第二步里：

- 单独展示的上卦图形
- 单独展示的下卦图形
- 组合后的本卦图形

三者必须严格联动。

也就是：

- 本卦上半必须和上卦单图完全一致
- 本卦下半必须和下卦单图完全一致

不要让 image2 在画“本卦”时重新脑补上半和下半。

如果上卦是 `艮 = 阳 / 阴 / 阴`，
那么本卦的上半三爻也必须还是 `阳 / 阴 / 阴`。

如果下卦是 `离 = 阳 / 阴 / 阳`，
那么本卦的下半三爻也必须还是 `阳 / 阴 / 阳`。

执行时可以把这条理解成：

- 先画上卦单图
- 先画下卦单图
- 本卦区域直接复制这两个三爻块
- 不要再让 image2 重新脑补一个“像是那个卦”的图
- 复制后还要再次检查：上卦确实有 3 条线，下卦确实有 3 条线，本卦确实有 6 条线

## 第四步必须体现

- 八卦五行对应
- 五行生克关系
- 体
- 用
- 体用关系
- 最终结论

## 五行生克图硬规则

第四步里的五行生克图，必须完整画出五个元素：

- 金
- 水
- 木
- 火
- 土

而且必须同时有：

- 相生链
- 相克链

不能少任何一个元素，不能只剩四个节点。

## 第三步必须体现

- 两数先相加
- 再除以 6
- 余数等于动爻
- 动爻按自下而上计数
- 动前这一爻是阴还是阳
- 动后这一爻变成什么

如果案例是 `73`，第三步必须是：

- `7+3=10`
- `10÷6，余4`
- `动爻：第四爻`

严禁把第三步画成两个数字直接相除。

同时要让用户能看出来：

- 动之前的上卦图形是什么
- 动之后为什么变成另一个卦
- 这个变化要能回到“八卦图形对照区”里核对

如果 `result.json` 里存在 `moving_line_change`：

- 必须直接使用其中的 `original_word`
- 必须直接使用其中的 `changed_word`
- 不允许 image2 自己猜“这一爻原本是阴还是阳”

如果 `qa_report.json` 已存在：

- 必须直接把“自下而上第几爻 = 自上而下第几条”当作硬约束
- 例如 `73` 这个案例里，第四爻动就必须画成“自上而下第3条高亮”

## 默认工作流

1. 读取 `result.json`
2. 核对确定性 SVG
3. 读取参考案例文档
4. 编写 image2 Prompt
5. 调用 `$imagegen`
6. 生成后人工复核
7. 若有错字、错卦、错动爻，明确判为不合格
8. 不合格时保留图片作为草稿，但标准结果仍以 SVG 为准
9. 画完以后，必须再交给 `meihua-diagram-qa` 校验

## Prompt 原则

- 把图当成“教学卡”
- 不要当成“玄幻海报”
- 中文要大、稳、清楚
- 卦线要准确
- 动爻高亮要明确
- 第二步和第四步的信息密度要高于装饰
- 五行图必须是完整五节点闭环，不允许漏掉任何一个元素
- 八卦图示必须严格遵守固定表，不允许模型自由发挥

## 输出纪律

- 默认同时保留：
  - 后台 SVG 校准版
  - 前台 image2 PNG 版
- 前台默认只展示 `image2_teaching_card.png`
- 不要把 SVG 当主结果返回
- 不要把 image2 错图当事实标准图
- 图画完不等于完成，必须通过 `meihua-diagram-qa`
